# Production hardening: rate limiting, SSE resilience, input validation

The change adds three layers of production-readiness hardening to the Horrorgeticon Ops server: an in-memory sliding-window rate limiter protecting auth and general API endpoints, SSE connection management with per-person limits and a stale-connection reaper, and systematic input validation (length caps, role allowlists, code pattern checks, JSON depth protection) across all mutation endpoints. The approach is deliberately zero-dependency, fitting the project's single-process, in-memory architecture. Graceful shutdown now drains SSE clients before snapshotting.

**Watch for:** The centralized body parsing in `main.js` bypasses the `jsonDepth` check that only lives in the now-unused `parseBody` (confirmed). Auth rate-limiting applies per-IP but on an event LAN behind NAT all crew shares one IP, which would lock everyone out after 10 failed logins (likely). The `RateLimiter` cleanup intervals are never destroyed on shutdown — minor leak but inconsistent with the otherwise careful cleanup (confirmed).

## High-level view

Rate limiting uses a custom `RateLimiter` class with a per-IP sliding window of timestamps. Auth endpoints get a stricter budget (10/5min) and the general API gets 500/min. The implementation interacts badly with shared-IP environments like an event LAN where all crew exits through one NAT address.

SSE resilience introduces a global cap (200 connections), a per-person cap (3, oldest evicted), and a background reaper that cleans up stale or socket-destroyed connections every 60 seconds. Graceful shutdown sends a `shutdown` event to all SSE clients before closing the HTTP server.

Input validation is applied at the module level: text fields get `.slice()` caps, roles are filtered against the canonical `ROLES` array, personal codes are regex-validated, scope objects are type-checked, and break durations are clamped to 1–120 minutes. Body size is enforced centrally at 256KB (8MB for CSV import). A `jsonDepth` function exists in `http.js` but is only reachable through `parseBody`, which is no longer called — the inline body parsing in `main.js` skips depth checking entirely.

## Rate limiting via sliding-window `RateLimiter`

Auth routes check the strict limiter first, then *also* hit the general limiter:

```js
if (isAuthRoute) {
  if (!authLimiter.allow(ip)) { throw ... }
}
if (!apiLimiter.allow(ip)) { throw ... }
```

Every login attempt costs two rate-limit tokens. An attacker hammering login burns through the general 500/min budget, potentially blocking legitimate data queries from the same IP. The fix is to skip the general limiter for auth routes (early-return after the auth check).

On a typical event LAN all crew traffic exits through one NAT IP. With `max: 10` over 5 minutes on the auth limiter, a single crew member fat-fingering their PIN 10 times locks out every other crew member's login for 5 minutes. On event night with 50+ actors checking in simultaneously, this is a realistic failure mode. Consider keying on `code + IP` or providing a management reset.

## SSE connection lifecycle and stale reaper

One asymmetry: `attach()` uses `now()` (the kernel's epoch-ms helper) for `client.since` but `Date.now()` for `lastWrite`. If these ever diverge, the reaper's cutoff comparison against `lastWrite` would misbehave. Currently `now()` wraps `Date.now()`, so this is latent rather than active.

## JSON depth check gap in centralized body parsing

`main.js` moved body parsing inline to enforce the configurable size limit:

```js
const raw = await readBody(req, bodyLimit);
// ...
body = JSON.parse(raw.toString('utf8'));
```

The `jsonDepth()` function lives in `http.js` inside the `parseBody` export. Since `main.js` imports `readBody` directly and never calls `parseBody`, depth protection is never applied to production requests. The smoke test for JSON depth bombs (`POST /api/people` with 55-level nesting) should fail under this code path — if it passes in CI, it may be exercising a stale server process or the test infrastructure has a different execution path. This warrants verification.

## Announcement scope validation gap

The `scope` validation in announcements checks `typeof scope !== 'object'` and validates `scope.type`, but doesn't validate `scope.mazeId` or `scope.positionIds` contents when `scope.type` is `'maze'` or `'positions'`. A client could pass `{ type: 'maze', mazeId: 'nonexistent' }` and `audiencePersonIds()` would return an empty Set — the announcement silently goes out to nobody. A guard on maze/position existence would prevent silent failures.

## Test coverage gaps

Not tested: the reaper actually cleaning up a stale connection (would require holding a connection open for 90s), the graceful shutdown event reaching a connected SSE client, or the 30s request timeout mechanism firing on a slow handler. The SSE global limit (503 at MAX_CLIENTS) is acknowledged as only indirectly verified.
